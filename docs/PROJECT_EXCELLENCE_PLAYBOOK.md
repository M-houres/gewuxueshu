# 项目升级作战手册（产品 / 开发 / 测试 / 运维 / 增长）

## 目标
把项目从“能跑”升级到“能稳定上线、能持续增长、能快速迭代”。

## 1. 产品设计视角
- 统一主流程：`选择平台 -> 上传正文 -> 提交确认`。
- 统一认知层级：每页只保留一个主动作，弱化次要信息。
- 统一视觉规范：步骤标题统一为大号加粗，关键状态（选中、警告、成功）颜色一致。

验收标准：
- 新用户首次进入 10 秒内能找到主入口。
- 三个功能页步骤命名一致且顺序一致。

## 2. 开发视角
- 抽离重复页面逻辑为组件（已完成登录/注册统一入口）。
- 部署与质量脚本统一放在 `scripts/`，避免口口相传。
- 改动必须可构建，可回滚，可审计。

验收标准：
- 前端构建通过。
- 关键改动有独立 commit，单次改动职责明确。

## 3. 测试视角
- 引入统一质量闸门：
  - Linux: `scripts/quality_gate.sh`
  - Windows: `scripts/quality_gate.ps1`
- 质量闸门至少覆盖：
  - 前端生产构建
  - 后端核心自动化测试（依赖满足时）

验收标准：
- 每次发布前执行一次质量闸门并保留结果。

## 4. 运维视角
- 使用单脚本部署：`scripts/deploy_aliyun.sh`。
- 脚本内置：
  - Git 克隆失败自动切换 codeload 下载
  - 自动生成关键密钥
  - 自动 docker compose 启动与健康检查
  - 可选 Docker 镜像加速配置

验收标准：
- 新机器 10 分钟内可拉起服务。
- 可重复执行，不依赖手工临时命令。

## 5. 增长与推广视角
- 登录前可浏览，提交时再登录，降低首访阻力。
- 在个人中心统一承接任务记录和积分流水，减少流失。
- 推广奖励可配置，并保留风控日志。

验收标准：
- 登录页到首个任务提交路径少于 3 次点击。
- 推广奖励策略后台可改且有审计。

## 6. 市场与用户视角
- 对外表述统一：学术场景、仿平台检测、可追溯结果。
- 对内后台统一：权限分层、配置在线生效、变更可追踪。
- 优先减少“用户要思考”的次数，而不是堆功能。

验收标准：
- 新用户无需阅读长文案即可完成首单。
- 运营同学不改环境变量也能完成日常配置。

## 发布前清单（建议每次上线执行）
1. 运行质量闸门。
2. 确认 `.env.prod` 关键密钥非默认值。
3. 执行部署脚本并通过健康检查。
4. 抽样验证三条用户路径：
   - 手机验证码登录并提交任务
   - 微信登录并提交任务
   - 后台修改配置并前台生效
